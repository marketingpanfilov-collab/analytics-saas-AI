# Аудит реализации DDA (Data-Driven Attribution)

**Дата:** 2025-03-06  
**Задача:** Анализ реализации «data-driven» атрибуции: формула, реальная модель, подтверждение, является ли это настоящей DDA. Изменения в коде не вносились.

---

## SECTION 1 — ПОИСК DDA ЛОГИКИ

### Где рассчитываются веса каналов

Единственное место расчёта весов для модели с названием **data_driven** — файл **`app/lib/attributionModels.ts`**, функция **`calculateDataDriven`** (строки 112–161).

- **`app/lib/assistedAttribution.ts`** — считает роли first_touch / assist / last_touch и агрегирует по каналам; **не** считает веса для DDA и не использует данные для обучения модели.
- **`app/lib/revenueAttributionMap.ts`** — использует результат `buildAssistedAttribution` (revenue_closed = last_touch, revenue_assisted); к DDA не обращается.
- **`app/lib/topAttributionPaths.ts`** — агрегирует пути (визиты → конверсия); весов не считает.
- **`app/lib/attributionFlow.ts`** — строит потоки на базе assisted attribution; DDA не используется.
- **`app/lib/attributionJourney.ts`** — собирает touchpoints по цепочкам и вызывает **`calculateAttributionModels`** (в т.ч. `calculateDataDriven`) для каждого journey; сам расчёт весов — в attributionModels.
- **`app/lib/budgetOptimizationInsights.ts`** — использует уже посчитанные **`attribution_models`** из journey (в т.ч. `data_driven`) для revenue по каналам и инсайтов; расчёт весов не выполняет.

**Итог:** веса каналов для «data_driven» считаются **только** в **`app/lib/attributionModels.ts`** в функции **`calculateDataDriven`**.

## SECTION 2 — ФОРМУЛА АТРИБУЦИИ

### Какая модель реализована

В коде явно указано: **«Simplified data-driven»** (комментарий к `calculateDataDriven`). Реализована **не** одна из классических моделей:

| Модель | Используется? |
|--------|----------------|
| **Position-based (40/40/20)** | Нет для data_driven; отдельно реализована в `calculatePositionBased` (40% first, 40% last, 20% middle). |
| **Linear** | Нет для data_driven; есть отдельно в `calculateLinear` (равные доли). |
| **Time decay** | Нет; временной фактор не используется. |
| **Markov chain** | Нет. |
| **Shapley value** | Нет. |
| **Probability model** | Нет. |
| **Custom heuristic** | **Да.** Фиксированные базовые веса по позиции + небольшие бонусы за наличие visit/registration после клика. |

### Точная формула «data_driven» в коде

Входные данные: массив **touchpoints** (типы: click, visit, registration, purchase) с полями type, source, platform, timestamp; **totalRevenue** — выручка journey.

1. Берутся только касания типа **click**, в порядке по времени: `getOrderedClicks(touchpoints)`.
2. Для каждого клика по индексу `i` (0 = первый, n-1 = последний) задаётся **базовый вес** `w`:
   - **n = 1:** `w = 1`.
   - **n = 2:** первый клик `w = 0.3`, второй `w = 0.7`.
   - **n ≥ 3:** первый `w = 0.3`, последний `w = 0.5`, каждый средний `w = 0.2 / (n - 2)`.
3. **Бонусы** (добавляются к `w`):
   - +0.05, если после этого клика и до следующего клика есть хотя бы один touchpoint типа **visit**.
   - +0.05, если после этого клика и до следующего клика есть хотя бы один touchpoint типа **registration**.
4. Нормализация: сумма весов по всем кликам приводится к 1 (`norm = 1 / sumW`), затем доля выручки канала = `totalRevenue * weight[i] * norm`.
5. Итог по source: для каждого канала суммируется вклад всех кликов с этим source (после нормализации).

**Формула в виде весов по позиции (до бонусов и нормализации):**

- 1 клик: [1].
- 2 клика: [0.3, 0.7].
- 3+ кликов: [0.3, 0.2/(n-2), …, 0.2/(n-2), 0.5] + бонусы visit/reg.

После добавления бонусов сумма весов может быть > 1, поэтому применяется нормализация.

## SECTION 3 — РАСЧЁТ ВЕСОВ КАНАЛОВ

### Пример: Meta → Google → Direct → Purchase

Допустим, три клика (Meta, Google, Direct), затем purchase с value = 100.

- n = 3.
- Базовые веса: first = 0.3, middle = 0.2/(3-2) = 0.2, last = 0.5.
- Для каждого клика проверяется: есть ли visit или registration между этим кликом и следующим. Если после Meta есть visit, у Meta +0.05; если после Google есть registration, у Google +0.05 и т.д.
- Допустим, бонусов нет. Веса: [0.3, 0.2, 0.5], сумма = 1.
- Распределение value = 100:
  - Meta   = 100 × 0.3 = **30**
  - Google = 100 × 0.2 = **20**
  - Direct = 100 × 0.5 = **50**

Если с бонусами (например, после Meta есть visit, после Google — registration):

- Meta: 0.3 + 0.05 = 0.35  
- Google: 0.2 + 0.05 = 0.25  
- Direct: 0.5  
- Сумма = 1.1 → norm = 1/1.1.  
- Meta = 100 × (0.35/1.1) ≈ **31.82**, Google ≈ **22.73**, Direct ≈ **45.45**.

**Веса фиксированные** в смысле: коэффициенты 0.3 / 0.5 / 0.2 и бонусы 0.05 заданы в коде и **не пересчитываются по данным** (нет обучения, нет оценки вклада каналов по истории конверсий).

## SECTION 4 — ДАННЫЕ ДЛЯ МОДЕЛИ

### Откуда берутся данные

Модель **не** обращается к БД напрямую. Она получает уже собранные **touchpoints** и **totalRevenue** для одного journey. Цепочка данных:

1. **attributionDebugger** строит цепочки из:
   - **redirect_click_events** (bq_click_id, traffic_source, traffic_platform, created_at, …),
   - **visit_source_events** (visitor_id, click_id, traffic_source, created_at, …),
   - **conversion_events** (visitor_id, click_id, user_external_id, value, …).
2. **attributionJourney.buildJourneysFromChains** группирует цепочки по identity (user_external_id, visitor_ids, click_ids), для каждой группы собирает **touchpoints** (click, visit, registration, purchase) с полями type, source, platform, timestamp, value и т.д.
3. Для каждого journey вызывается **calculateAttributionModels(touchpoints, revenue)**; внутри вызывается **calculateDataDriven(touchpoints, totalRevenue)**.

### Какие поля реально использует calculateDataDriven

- **type** — отбираются только touchpoints с type === "click".
- **source** — нормализация в метку канала (meta, google, tiktok, yandex, unknown); по ней агрегируется revenue.
- **timestamp** — порядок touchpoints уже задан при сборке; в самой функции порядок кликов не пересортировывается (используется порядок в массиве).
- **tpIndex** — позиция клика среди всех touchpoints; нужна для проверки «есть ли visit/registration между этим и следующим кликом».

**Не используются:** visitor_id, click_id, session_id, user_external_id, created_at как таковые — они использованы на этапе построения цепочек и journey, но не внутри формулы DDA. Модель работает только с упорядоченным списком кликов и типами событий между ними (visit, registration).

## SECTION 5 — УЧЁТ ПОРЯДКА КАСАНИЙ

**Да.** Позиция канала в пути напрямую влияет на вес:

- Первый клик всегда получает базовый вес 0.3 (или 1 при одном клике).
- Последний клик — 0.5 при n ≥ 3 (или 0.7 при n = 2).
- Средние — по 0.2/(n-2) при n ≥ 3.

То есть пути **Meta → Google → Direct** и **Direct → Meta → Google** дадут разное распределение для одних и тех же каналов: в первом случае Meta = 0.3, Direct = 0.5; во втором Direct = 0.3, Meta = 0.5 (при отсутствии бонусов).

---

## SECTION 6 — УЧЁТ ВРЕМЕНИ

**Нет.** В `calculateDataDriven` нет временного фактора:

- Нет расстояния до конверсии (time to conversion).
- Нет затухания по времени (time decay).
- Используется только порядок кликов (первый / средние / последний), а не абсолютное время или интервалы.

Время влияет только косвенно: порядок touchpoints при сборке journey строится по timestamp, поэтому «первый/последний» — это первый/последний по времени.

---

## SECTION 7 — УЧЁТ ЧАСТОТЫ КАНАЛОВ

**Нет.** Модель не анализирует частоту появления каналов в успешных путях:

- Нет агрегации по множеству конверсий или путей.
- Нет пересчёта весов в зависимости от того, как часто канал встречается в конверсиях.
- Веса заданы фиксированно по позиции (0.3 / 0.5 / 0.2) и бонусам visit/reg.

То есть «если Meta часто в успешных путях» никак не увеличивает долю Meta в этой модели.

---

## SECTION 8 — ПЕРЕСЧЁТ МОДЕЛИ

**При каждом запросе.** Модель не обучается и не кэшируется:

- Нет сохранённых весов по каналам или по проекту.
- Нет дневного/периодического пересчёта.
- При каждом запросе к API (attribution-journeys, budget-optimization-insights) заново строятся цепочки → journey → для каждого journey вызывается `calculateAttributionModels` → в том числе `calculateDataDriven` с теми же правилами.

«Пересчёт модели» в смысле обучения весов по данным отсутствует — пересчитывается только применение одной и той же эвристики к новым данным.

---

## SECTION 9 — DATA REQUIREMENTS

В коде **нет** явных минимальных порогов:

- Нет проверки «минимум N конверсий» или «минимум N путей» для включения data_driven.
- Если кликов 0 или totalRevenue ≤ 0, функция возвращает пустой объект.
- При одном клике ему отдаётся 100% выручки.

Пороги могут быть на уровне API (период, лимиты выборки), но не внутри самой формулы DDA.

---

## SECTION 10 — РЕАЛЬНО ЛИ ЭТО DDA

**Нет. Это не настоящая Data-Driven Attribution.**

Причины:

1. **Нет обучения на данных** — веса 0.3 / 0.5 / 0.2 и бонусы 0.05 заданы в коде и не выводятся из истории конверсий.
2. **Нет вероятностной/статистической модели** — нет оценки вклада канала в вероятность конверсии (например, логистическая регрессия, вероятность перехода и т.п.).
3. **Нет методов, характерных для DDA в индустрии:**
   - нет Shapley value;
   - нет Markov chain (переходы между состояниями);
   - нет ML-модели, обученной на конверсиях/неконверсиях.
4. Реализация — **кастомная эвристика**: позиционная схема (усиленный last touch: 0.3 first, 0.5 last, 0.2 middle) плюс небольшие бонусы за visit/registration после клика.

В отчёте корректно называть её **упрощённой позиционно-эвристической моделью** или **«simplified data-driven» (эвристика)**, а не настоящей DDA.

---

## SECTION 11 — ПРОИЗВОДИТЕЛЬНОСТЬ

- **SQL:** только на этапе построения цепочек (attributionDebugger) и при загрузке visit/conversion/click; в `calculateDataDriven` запросов к БД нет.
- **Расчёт весов:** полностью **in-memory**: один проход по кликам, подсчёт весов, нормализация, распределение totalRevenue по source. Сложность O(n) по числу кликов в journey.
- **Кэширование:** результатов data_driven или весов по каналам нет; при каждом запросе расчёт заново.

Тяжёлая часть — построение цепочек и journey по большому объёму событий; сам вызов `calculateDataDriven` лёгкий.

## SECTION 13 — ОГРАНИЧЕНИЯ

- Нет **time decay**; время до конверсии не учитывается.
- Нет **вероятностной модели** и оценки вклада каналов из данных.
- Нет **ML** и обучения весов на конверсиях/неконверсиях.
- Нет **Shapley value**, **Markov chain**, других классических DDA-методов.
- Веса **фиксированы** (0.3 / 0.5 / 0.2 + бонусы 0.05); не зависят от объёма или качества данных.
- Учитываются только **клики** (тип "click") в touchpoints; визиты без клика в этой модели не получают доли выручки.
- Модель считается **на уровне journey** (один пользователь/путь); агрегация по каналам — простая сумма вкладов по всем journey.

## SECTION 14 — ИТОГОВЫЙ ВЫВОД

### 1. Реальная модель атрибуции

**Упрощённая позиционно-эвристическая модель**, в коде названная «Simplified data-driven»: фиксированные базовые веса по позиции клика (first / middle / last) плюс небольшие бонусы за наличие visit или registration после клика. Не является настоящей Data-Driven Attribution (нет обучения на данных, нет вероятностной/игровой модели).

### 2. Используемая формула

- Базовые веса по позиции: первый клик 0.3, последний 0.5, каждый средний 0.2/(n-2); при n=1 вес 1; при n=2 веса 0.3 и 0.7.
- К весу каждого клика добавляется +0.05 при наличии visit после этого клика до следующего клика и +0.05 при наличии registration на том же отрезке.
- Нормализация весов к сумме 1; доля выручки канала = totalRevenue × (нормализованный вес по каждому клику), с группировкой по source.

### 3. Используемые данные

Touchpoints одного journey (тип, source, порядок, наличие visit/registration между кликами) и totalRevenue этого journey. Исходные данные — redirect_click_events, visit_source_events, conversion_events (на этапе построения цепочек и journey), но внутри формулы DDA используются только упорядоченные клики и типы событий между ними.

### 4. Сильные стороны реализации

- Простая, детерминированная формула; быстрый in-memory расчёт.
- Учитывается порядок касаний и контекст (visit/registration после клика).
- Сравнение с first/last touch в Debugger и Budget Insights даёт понятные инсайты (пере/недооценка канала).
- Не требует больших объёмов данных и не имеет явных минимальных порогов в самой функции.

### 5. Ограничения модели

- Не является настоящей DDA: веса не выводятся из данных.
- Нет time decay, частоты каналов в путях, вероятностной или ML-модели.
- Фиксированные коэффициенты могут не соответствовать реальной важности каналов в конкретном продукте.
- В расчёте участвуют только клики; визиты без клика не получают доли в этой модели.
