# Аудит раздела /app/ltv — технический отчёт

**Дата:** 2025  
**Задача:** анализ текущей логики, источников данных и формул без изменений кода.

---

## 1. Общая архитектура /app/ltv

- **Страница:** `app/app/(with-sidebar)/ltv/page.tsx` — единственный клиентский компонент страницы (отдельного page wrapper нет, только layout).
- **Данные:** один источник — `GET /api/ltv?project_id=&start=&end=&cohort_month=`. Параметры:
  - `project_id` — из `useSearchParams()` (обязателен);
  - `start`, `end` — фиксированный диапазон «последние 3 месяца» (вычисляется при монтировании, не меняется пользователем);
  - `cohort_month` — выбранная когорта (YYYY-MM), опционально.
- **Компоненты:**
  - `LtvChart` (`app/components/LtvChart.tsx`) — график LTV/ARPU по точкам из API.
  - `CohortHeatmap` (`app/components/CohortHeatmap.tsx`) — тепловая карта когорт; данные из API, режимы: percent / users / money (пересчёт на клиенте из percent + cohortSizes и фиксированного ARPU по месяцам).
- **Фильтры UI:** «Канал» (channel) и «Когорта» (cohort month) — только на клиенте; **в API параметр канала не передаётся**, фильтрация по каналу не применяется.

## 2. Источники данных

| Источник | Использование в /app/ltv |
|----------|--------------------------|
| **conversion_events** | Единственная таблица для покупок: все purchase за период `[start, end]` по `event_time`. Поля: id, event_time, created_at, user_external_id, visitor_id, value. |
| **project_monthly_plans** | Только для выбранного cohort_month: чтение `repeat_sales_budget` (бюджет на повторные продажи). |
| **getCanonicalSummary()** (daily_ad_metrics) | Итоговый spend за период; возвращается в kpi.spend, на UI не выводится. |
| **Моки / демо** | Нет отдельного demo mode. При отсутствии project_id показывается «Выберите проект»; при ошибке — текст ошибки. Все числовые блоки питаются из API. |
| **Жёстко заданные значения** | На странице: `retentionMoM = 0`, `revenueMoM = 0`, `payingShare = 0` (в API). Для режима «Оборот (₸)» в когортах на клиенте: массив `arpuByMonthIndex = [8000, 6200, 5400, 4800, 4300, 3900, 3600]` — условный ARPU по месяцам M0…M6. |

**Итог:** источник правды для покупок и выручки — **conversion_events**. Для бюджета на повторные продажи — **project_monthly_plans**. visit_source_events, redirect_click_events и campaign_intent в текущей логике LTV не используются.

## 3. First / repeat purchase logic

**Где считается:** только в `app/api/ltv/route.ts`, в памяти после выборки покупок.

**User key (идентификатор пользователя):**
```ts
user_key = user_external_id?.trim() || visitor_id?.trim() || id
```
- Приоритет: `user_external_id` → `visitor_id` → `id` (uuid события).
- Если оба `user_external_id` и `visitor_id` пустые, используется `id` события: такая покупка всегда считается одной (одна «личность» = одна покупка), повторной быть не может.

**Определение first / repeat:**
1. Все покупки за период запрашиваются из conversion_events по `event_time` между start и end, сортировка по `event_time` asc.
2. События группируются по `user_key`.
3. Внутри каждого пользователя покупки сортируются по `event_time`.
4. Первая по времени покупка (индекс 0) — **first purchase**, остальные — **repeat purchase**.

**Риски:**
- Одна и та же покупка не может учитываться и как first, и как repeat: разбиение по индексу внутри пользователя взаимоисключающее.
- Покупки **вне** выбранного периода не участвуют: если у пользователя первая покупка была до `start`, в этом периоде все его покупки в LTV будут считаться repeat. То есть first/repeat — только **в рамках выбранного окна дат**, а не глобально за всё время.
- При отсутствии user_external_id и visitor_id использование `id` даёт «одноразового» пользователя на каждую покупку: завышается число «уникальных» и first, занижается repeat.

## 4. Формулы основных метрик

**Обозначения:**  
- По пользователям: группировка по user_key, сортировка по event_time.  
- `byUser` — Map<user_key, list of { value, eventTime }>.  
- Период: `[start, end]`.

| Метрика | Формула |
|--------|---------|
| **total_revenue** | Сумма всех `value` по всем покупкам в периоде. |
| **first_purchase_count** | Число пользователей (размер byUser), у которых в периоде есть хотя бы одна покупка (для каждой группы первая по времени считается first). Фактически: число «первых» покупок в периоде = число уникальных user_key. |
| **repeat_purchase_count** | Число покупок с индексом > 0 внутри каждого пользователя (сумма по всем пользователям). |
| **repeat_revenue** | Сумма `value` по всем покупкам с индексом > 0 внутри каждого пользователя. |
| **unique_purchasers** | `byUser.size`. |
| **revenueMi** | = total_revenue за период. |
| **arpuMi** | total_revenue / unique_purchasers (или 0). |
| **usersM0** | Число пользователей, у которых **дата первой покупки** (в рамках всех их покупок в периоде) попадает в месяц cohort_month (YYYY-MM). То есть когорта = «впервые купили в этом месяце». |
| **usersMi** | Если выбран cohort_month: число пользователей когорты cohort_month, у которых есть хотя бы одна покупка в **последнем** месяце диапазона M0…M6 (т.е. M6 для выбранной когорты). Если cohort_month не выбран: unique_purchasers. |
| **activeUsersMi** | Если cohort_month выбран: число пользователей когорты, совершивших покупку в M0 (первый месяц когорты). Иначе: unique_purchasers. |
| **retentionPct** | usersM0 > 0 ? (m0Active / usersM0) * 100 : 0. По сути доля когорты, оставшейся «активной» в M0. |
| **ltvCum** | Для выбранной когорты: средний накопленный revenue на пользователя к D90 (последний горизонт). LTV по горизонтам D1…D90 считаются как среднее по когорте суммы value пользователя в окне [first_purchase_time, first_purchase_time + D дней]. Без когорты или при usersM0=0 — 0. |
| **lineData (LTV/ARPU)** | Точки D1, D7, D14, D30, D60, D90. Для каждой: ltv = среднее по когорте накопленной выручки до этого горизонта; arpu в API дублирует ltv (то же значение). |
| **ltvXUsers** | ltvCum * cohortSize (cohortSize = usersM0 при выбранной когорте, иначе unique_purchasers). |
| **payingShare** | Всегда 0 (в API). Задумывалось как покупатели/регистрации за период — регистрации не запрашиваются. |
| **retentionMoM, revenueMoM** | Всегда 0 (в API не считаются). |
| **budget_for_repeat_sales** | project_monthly_plans.repeat_sales_budget для (project_id, year, month) из cohort_month. Если cohort_month не передан или плана нет — null. |
| **CPR** | budget_for_repeat_sales / repeat_purchase_count, если оба заданы и repeat_purchase_count > 0; иначе null. Infinity/NaN не отдаётся. |
| **spend** | getCanonicalSummary(project_id, start, end) → totals.spend. В ответе есть, на UI не выводится. |
| **cohortRows** | Для последних 5 месяцев, в которых есть «первые покупки» пользователей: по каждой когорте (месяц первой покупки) — 7 значений (M0…M6): доля пользователей когорты, совершивших хотя бы одну покупку в соответствующем месяце, в % (0–100). |
| **cohortSizes** | Для каждой когорты в cohortRows — число пользователей с первой покупкой в этом месяце. |

## 6. Согласованность с /app dashboard

- **Дашборд (KPI):** покупки и выручка считаются из **conversion_events** по **created_at** в диапазоне [start, end], с опциональной фильтрацией по source (visit_source_events + source_classification).
- **LTV:** те же **conversion_events**, но фильтр по **event_time** (не created_at) и **без** фильтра по каналу/source.

Возможные расхождения:
1. **created_at vs event_time:** если событие записано с задержкой, дата в LTV (event_time) и в дашборде (created_at) может отличаться — разные периоды попадания.
2. **Фильтр по каналу:** на дашборде можно смотреть по источникам; на LTV выбор «Канал» не влияет на запрос — всегда «все покупки».
3. **First/repeat на дашборде:** основной дашборд не показывает разбиение first/repeat; LTV считает его только по окну [start, end], а не за всё время — прямого сравнения «repeat count» с дашбордом нет.

Spend: на LTV приходит из getCanonicalSummary (тот же источник, что и для дашборда), но на экране LTV не отображается.

## 7. Текущие риски и несоответствия

1. **First/repeat только в периоде:** логика «первая покупка в жизни» не используется — «первая в периоде» может быть повторной в реальности, если первая была до start.
2. **User key fallback на event id:** при отсутствии user_external_id и visitor_id каждая покупка считается отдельным «пользователем» — завышение уникальных и first, занижение repeat.
3. **Лимит 50k событий:** пагинация по 1000, максимум 50 страниц; при большом объёме данные обрезаются без предупреждения.
4. **Доля платящих и MoM:** не считаются (всегда 0); для payingShare нужны регистрации за период.
5. **Канал на LTV не применяется:** выбор в селекторе не передаётся в API.
6. **Подпись в CohortHeatmap:** текст «Заглушка · позже подключим реальные данные» не соответствует факту (данные приходят из API).
7. **Режим «Оборот» в когортах:** использует жёсткий массив ARPU по месяцам — не из данных.
8. **Дата диапазона:** start/end задаются один раз при загрузке (последние 3 месяца), пользователь не может сменить период.

## 8. Retention / CPR / бюджет на повторные продажи (текущее состояние)

- **Retention marker (campaign_intent):** в LTV не используется. В conversion_events поле может быть, но API LTV его не читает и не фильтрует по нему.
- **Бюджет на повторные продажи:** только из плана (`project_monthly_plans.repeat_sales_budget`) на выбранный месяц когорты. Фактического «retention spend» из рекламы/кликов в расчёте нет.
- **CPR:** считается как план по повторным продажам / фактическое число повторных покупок за период. Не привязан к retention-трафику или retention spend.
- **Repeat purchase rate / retention revenue share:** явно на экране не выводятся; при необходимости выводятся из repeat_purchases_count, repeat_revenue и total (формулы в API есть).

Итог: экран опирается на концепцию «repeat purchase» (вторая и далее покупки по user_key в периоде), без разделения по retention/acquisition и без использования retention-marked spend.

## 10. Что дорабатывать дальше (без изменений в этой задаче)

1. Уточнить first/repeat: считать первую покупку **за всё время** (глобально), а не только в выбранном периоде.
2. Учитывать campaign_intent в LTV: фильтрация/агрегаты по retention-трафику, при наличии — retention spend и CPR по факту.
3. Считать payingShare при наличии регистраций; по желанию — retentionMoM, revenueMoM.
4. Передавать фильтр канала в API и фильтровать покупки по источнику (или явно документировать «все каналы»).
5. Добавить выбор периода (start/end) на UI.
6. Убрать или переформулировать подпись «Заглушка» в CohortHeatmap; режим «Оборот» при возможности перевести на факт из данных.
7. Ограничение 50k событий: либо предупреждение, либо агрегация на стороне БД (RPC/materialized view).

**Итог:** раздел /app/ltv полностью питается из API `/api/ltv`, который строит метрики на основе **conversion_events** (event_time, user_external_id, visitor_id, value) и **project_monthly_plans**. Моков в виде отдельных массивов нет; часть показателей зафиксирована нулями или константами. First/repeat определяется только внутри выбранного периода по user_key (user_external_id || visitor_id || id). Для дальнейшего учёта retention нужны доработки: использование campaign_intent и, при наличии данных, фактического retention spend.
